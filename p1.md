Prompt 1: order-sync-library
Create a Spring Boot 3.2 shared library called "order-sync-library" for syncing pharmacy order data from edge sites to GCP via Kafka.

## Context
- 40 pharmacy sites, each with 20-30 Spring Boot services using JdbcTemplate
- All tables partitioned by site_id
- Orders flow through events: RECEIVED → INITIALIZED → ON_HOLD → RELEASED → INDUCTED → PICKED → CHECKED → PACKED
- Each event touches different tables (e.g., RECEIVED touches 5 tables, PICKED touches 13)
- This library is published as a JAR to internal Maven repo. Services add it as dependency, add 3 YAML properties, and call one method.

## Hybrid Publish Flow (critical design — follow exactly)

1. Business service calls `syncService.flagSync(orderId, OrderEvent.ORDER_PICKED)` inside its @Transactional method
2. flagSync() uses @Transactional(propagation = MANDATORY) — ensures it runs inside caller's transaction
3. Writes to `order_sync_outbox` table in the SAME transaction (safety net)
4. Uses `RETURNING id` to get the outbox entry ID
5. Registers `TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() { afterCommit() {...} })`
6. After the business transaction commits, afterCommit() fires:
   a. Calls BundleBuilder.buildBundle() to fetch data (data is now committed and visible)
   b. Publishes to Kafka immediately via SyncKafkaProducer
   c. If Kafka succeeds → marks outbox entry as SYNCED
   d. If Kafka fails or any error → logs warning, outbox stays PENDING (scheduler handles it)
   e. attemptImmediatePublish() NEVER throws — all errors caught and logged
7. Background OutboxScheduler polls every 3 seconds for PENDING entries (fallback)
   - Claims batch using FOR UPDATE SKIP LOCKED
   - Builds bundle, publishes to Kafka, marks SYNCED
   - This rarely activates — only when Kafka is down or immediate publish failed

## Bundle Format (critical — consumer depends on this exact format)

The bundle includes metadata (column names, types, primary keys) and row data where ALL values are cast to TEXT by source PostgreSQL. This eliminates Java type conversion on the consumer side.
```json
{
  "orders": {
    "_meta": {
      "columns": ["id","site_id","order_id","status","priority","created_at"],
      "types": ["bigint","character varying","character varying","character varying","integer","timestamp with time zone"],
      "pk": ["id","site_id"]
    },
    "rows": [
      ["1","SITE_001","ORD-500","PICKED","1","2026-02-22 10:00:00+00"]
    ]
  },
  "rx": {
    "_meta": {
      "columns": ["id","site_id","order_id","drug_name","quantity","status","created_at"],
      "types": ["bigint","character varying","character varying","character varying","integer","character varying","timestamp with time zone"],
      "pk": ["id","site_id"]
    },
    "rows": [
      ["100","SITE_001","ORD-500","Lisinopril","30","ACTIVE","2026-02-22 10:00:00+00"],
      ["101","SITE_001","ORD-500","Metformin","60","ACTIVE","2026-02-22 10:00:00+00"]
    ]
  },
  "rx_staging": {
    "_meta": {
      "columns": ["id","site_id","order_id","status"],
      "types": ["bigint","character varying","character varying","character varying"],
      "pk": ["id","site_id"]
    },
    "rows": []
  }
}
```

Key design: every value in "rows" is a TEXT string formatted by source PostgreSQL using `::text` cast. Null values are JSON null. The consumer sends these as plain strings with `?::type` casts in SQL — target PostgreSQL parses its own text format. Zero Java type conversion needed.

## BundleBuilder — How to Fetch Data

BundleBuilder builds one SQL query per table (not one giant json_build_object for all tables, because we need per-table metadata).

For each table, it builds a query like this:
```sql
SELECT json_build_object(
  '_meta', json_build_object(
    'columns', '["id","site_id","order_id","status","priority","created_at"]'::jsonb,
    'types', '["bigint","character varying","character varying","character varying","integer","timestamp with time zone"]'::jsonb,
    'pk', '["id","site_id"]'::jsonb
  ),
  'rows', COALESCE((
    SELECT jsonb_agg(row_array)
    FROM (
      SELECT jsonb_build_array(
        t."id"::text,
        t."site_id"::text,
        t."order_id"::text,
        t."status"::text,
        t."priority"::text,
        t."created_at"::text
      ) AS row_array
      FROM "orders" t
      WHERE t.order_id = ? AND t.site_id = ?
    ) sub
  ), '[]'::jsonb)
)::text
```

This query is built dynamically. BundleBuilder caches table metadata (columns, types, pk) from information_schema and pg_index on first access per table using ConcurrentHashMap.

The metadata query for columns:
```sql
SELECT column_name, data_type FROM information_schema.columns
WHERE table_name = ? AND table_schema = 'public' ORDER BY ordinal_position
```

The metadata query for primary keys:
```sql
SELECT a.attname FROM pg_index i
JOIN pg_attribute a ON a.attrelid = i.indrelid AND a.attnum = ANY(i.indkey)
WHERE i.indrelid = ?::regclass AND i.indisprimary
ORDER BY array_position(i.indkey, a.attnum)
```

BundleBuilder iterates through the tables for the event, calls the per-table query, and assembles the full JSON bundle as a string.

## Event-to-Table Mappings (Database-Driven)

Stored in database table `sync_event_table_config`, NOT hardcoded. Cached in memory using AtomicReference, refreshed every 5 minutes via @Scheduled. If DB refresh fails, keep cached values.

Seed data:
- ORDER_RECEIVED: orders, patient, rx, order_details, rx_details
- ORDER_INITIALIZED: orders, rx, order_details, rx_details, order_comment
- ORDER_ON_HOLD: orders, rx, order_history, order_hold_exclusion, order_comment
- ORDER_RELEASED: same as ON_HOLD
- ORDER_INDUCTED: orders, order_history, tote, tote_history
- ORDER_PICKED: orders, order_details, order_history, rx, rx_staging, rx_loc, rx_history, container, container_content, container_history, container_content_history, tote, tote_history
- ORDER_CHECKED: same as PICKED
- ORDER_PACKED: same as PICKED

```json
{

  "siteId": "SITE_001",
  "eventType": "ORDER_PICKED",
  "bundleJson": "{...the bundle JSON above as escaped string...}",
  "createdAt": "2026-02-22T10:30:05Z",
  "outboxId": 42,
  "retryCount": 0
}
```
Topic: "order-sync-events". Key: {siteId}_{orderId}.
