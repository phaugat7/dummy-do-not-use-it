# Prompt: Implement Transactional Outbox Pattern with Async Kafka Publisher

## Context

I have an existing application that syncs data between an Edge database and GCP. I need to implement a **Transactional Outbox Pattern** so that every data mutation (INSERT, UPDATE, DELETE) performed by my APIs is captured as an event, stored locally in an outbox table, and propagated to Kafka ‚Äî all without degrading API performance.

The architecture must support **near-real-time** sync. Events should reach Kafka within milliseconds in the happy path, with a scheduler acting as a safety net for failures only.

---

## What I Need You to Implement

### 1. Outbox Events Table

Create a database migration for an `outbox_events` table with the following columns:

| Column | Type | Description |
|--------|------|-------------|
| `id` | BIGINT / AUTO_INCREMENT | Primary key |
| `event_id` | UUID | Unique identifier for deduplication on the consumer side |
| `aggregate_type` | VARCHAR(255) | Entity/table name (e.g., `Order`, `Customer`, `Product`) |
| `aggregate_id` | VARCHAR(255) | The ID of the entity that changed |
| `event_type` | VARCHAR(100) | `INSERT`, `UPDATE`, or `DELETE` |
| `payload` | TEXT / JSON | The full serialized entity state (or diff) after the change |
| `status` | VARCHAR(20) | `PENDING`, `SENT`, `FAILED` |
| `retry_count` | INT | Default 0, incremented on each failed attempt |
| `max_retries` | INT | Default 5 |
| `created_at` | TIMESTAMP | When the event was created |
| `sent_at` | TIMESTAMP | When the event was successfully published to Kafka (nullable) |
| `error_message` | TEXT | Last error message if publish failed (nullable) |

Add indexes on:
- `status` + `created_at` (for the scheduler query)
- `event_id` (unique index, for deduplication)
- `aggregate_type` + `aggregate_id` (for querying events by entity)

### 2. DataSource Proxy / Interceptor ‚Äî Outbox Insert

Using the existing DataSource `buildProxy` pattern (or equivalent interceptor like `afterQuery`, Hibernate event listener, JPA `@EntityListeners`, or framework-specific hooks):

- **After** a successful write operation (INSERT, UPDATE, DELETE), **within the same database transaction**, insert a row into `outbox_events` with `status = 'PENDING'`.
- The outbox INSERT must be part of the same transaction as the original write. If the transaction rolls back, the outbox row must also roll back. This guarantees atomicity ‚Äî no phantom events, no lost events.
- Capture the following in the payload:
  - The entity/table name
  - The entity ID
  - The operation type
  - The full entity state after mutation (for INSERT/UPDATE) or the entity ID (for DELETE)
  - A timestamp
- Generate a UUID `event_id` for each outbox row.

**Important constraints:**
- Do NOT publish to Kafka inside the transaction.
- Do NOT block the API response waiting for Kafka.
- Keep the interceptor lightweight ‚Äî it should only do the one extra INSERT.

---

### 3. Async Kafka Publisher ‚Äî Inline (Fire-and-Forget)

Immediately **after** the database transaction commits (not inside it), trigger an **asynchronous, non-blocking** Kafka publish:

```
TRANSACTION COMMIT
‚Üí Return API response to caller immediately
‚Üí In background (async / non-blocking):
    ‚Üí Publish the event to Kafka topic
    ‚Üí On success: UPDATE outbox_events SET status = 'SENT', sent_at = NOW() WHERE event_id = ?
    ‚Üí On failure: Log warning, do nothing (scheduler will retry later)
```

**Key rules:**
- The Kafka publish must be **fire-and-forget** from the API's perspective. The API response must NOT wait for Kafka.
- Use async mechanisms appropriate to the framework (e.g., `CompletableFuture` in Java/Spring, `Promise`/`async` in Node.js, goroutines in Go).
- If Kafka publish fails, do NOT throw or propagate the error to the API caller. Just log it. The scheduler will catch it.

---

### 4. Scheduler ‚Äî Safety Net for Failed/Missed Events

Implement a background scheduled job that runs every **30‚Äì60 seconds**:

```sql
SELECT * FROM outbox_events
WHERE status = 'PENDING'
  AND created_at < NOW() - INTERVAL '10 SECONDS'
  AND retry_count < max_retries
ORDER BY created_at ASC
LIMIT 100;
```

**The 10-second delay is critical** ‚Äî it prevents the scheduler from racing with the inline async publisher. Without this buffer, the scheduler could pick up a record that the inline publisher is already handling, causing unnecessary duplicate publishes.

For each record:
1. Publish to Kafka.
2. On success: `UPDATE status = 'SENT', sent_at = NOW()`
3. On failure: `UPDATE retry_count = retry_count + 1, error_message = '<error>'`
4. If `retry_count >= max_retries`: `UPDATE status = 'FAILED'` (and optionally alert/notify)

Use batch processing where possible ‚Äî fetch N records, publish them in a batch or concurrently, then batch-update their statuses.

---

### 5. Cleanup Job

Implement a daily scheduled job (cron) that deletes old completed events:

```sql
DELETE FROM outbox_events
WHERE status = 'SENT'
  AND sent_at < NOW() - INTERVAL '48 HOURS';
```

Optionally, also handle FAILED events:
- Move them to a dead-letter table for investigation, OR
- Alert on them and delete after 7 days.

---

### 6. Kafka Topic Structure

Use a topic naming convention like:

```
edge.outbox.{aggregate_type}
```

For example: `edge.outbox.Order`, `edge.outbox.Customer`

Or a single unified topic if preferred:

```
edge.outbox.events
```

The message key should be `aggregate_type:aggregate_id` to ensure ordering per entity.

The message value should be a JSON envelope:

```json
{
  "eventId": "uuid-here",
  "aggregateType": "Order",
  "aggregateId": "12345",
  "eventType": "UPDATE",
  "payload": { /* full entity state */ },
  "timestamp": "2026-03-22T10:30:00Z"
}
```

---

### 7. Idempotency on the Consumer Side (GCP)

The GCP consumer MUST deduplicate based on `eventId`. Since both the inline publisher and the scheduler could potentially send the same event, the consumer should:

- Maintain a table or cache of processed `eventId` values.
- Before processing, check if `eventId` has already been handled.
- Skip duplicates silently.

## Non-Functional Requirements

- **API latency impact**: The only synchronous overhead should be one extra INSERT within an existing transaction (microseconds, not milliseconds).
- **Kafka unavailability**: Must NOT impact API availability. Events queue in the outbox and are published when Kafka recovers.
- **Ordering**: Events for the same entity should be ordered by `created_at`. Use `aggregate_id` as the Kafka message key to ensure partition-level ordering.
- **At-least-once delivery**: The system guarantees at-least-once delivery. The consumer must be idempotent.
- **Observability**: Add metrics/logs for: events created per minute, events published per minute, current PENDING backlog size, FAILED event count, average time from PENDING to SENT.
