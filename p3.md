I have an existing order-sync-library (Spring Boot 3.2 shared JAR) that publishes order data to Kafka. I need to refactor the BundleBuilder component to handle tables that have different foreign key relationships.

## The Problem

Not all tables have an order_id column. Tables are grouped by their foreign key:

- ORDER-level tables (have order_id + site_id): orders, patient, order_details, order_history, order_comment, order_hold_exclusion, tote, tote_history, rx, container
- RX-level tables (have rx_id, NO order_id): rx_details, rx_staging, rx_loc, rx_history
- CONTAINER-level tables (have container_id, NO order_id): container_content, container_history, container_content_history

The "rx" and "container" tables are ORDER-level because they have order_id. But their child tables (rx_details, container_content, etc.) only have rx_id or container_id.

## The Solution: 3-Phase Fetch Chain with Strategy Pattern

Fetch data in 3 phases, each handled by a separate fetcher strategy class:

Phase 1: OrderGroupFetcher fetches all ORDER-level tables using WHERE order_id = ? AND site_id = ?
         This includes the "rx" and "container" tables.
         After fetching, extract rx.id values → these become rx_ids for Phase 2.
         After fetching, extract container.id values → these become container_ids for Phase 3.

Phase 2: RxGroupFetcher fetches all RX-level tables using WHERE rx_id IN (?, ?, ...)
         Only runs if the event has RX-group tables AND Phase 1 found rx records.

Phase 3: ContainerGroupFetcher fetches all CONTAINER-level tables using WHERE container_id IN (?, ?, ...)
         Only runs if the event has CONTAINER-group tables AND Phase 1 found container records.

Total database round trips: 1-3 depending on the event (not 14 individual queries).

## Class Design
BundleBuilder (orchestrator)
│ orchestrates 3 phases
├── OrderGroupFetcher    → fetchBySingleKey(tables, "order_id", orderId, siteId)
├── RxGroupFetcher       → fetchByMultipleIds(tables, "rx_id", rxIds)
└── ContainerGroupFetcher → fetchByMultipleIds(tables, "container_id", containerIds)
│
└── all extend TableGroupFetcher (abstract base class)

### TableGroupFetcher (abstract base class)

This handles ALL the heavy lifting. Subclasses are tiny.

**Key methods:**

`fetchBySingleKey(List<String> tables, String keyColumn, String keyValue, String siteId)`
- Builds ONE json_build_object query for all tables in the list
- WHERE clause: `t."order_id" = ? AND t."site_id" = ?`
- Returns Map<String, String> of tableName → table JSON

`fetchByMultipleIds(List<String> tables, String keyColumn, List<String> keyIds)`
- Builds ONE json_build_object query for all tables in the list
- WHERE clause: `t."rx_id" IN (?, ?, ?)`
- If keyIds is empty, returns empty results for all tables (no query executed)
- Returns Map<String, String> of tableName → table JSON

`extractIds(String tableJson, String idColumn)`
- Takes a single table's JSON result: {"cols":["id","site_id",...], "rows":[["100",...],["101",...]]}
- Finds the position of idColumn in the cols array
- Extracts that position's value from every row
- Returns List<String> of ID values

**SQL query format** (generated by base class for both fetch methods):
For each table, the query fetches column names as a "cols" array and all row values cast to ::text:
```sql
SELECT json_build_object(
  'orders', json_build_object(
    'cols', '["id","site_id","order_id","status","priority","created_at"]'::jsonb,
    'rows', COALESCE((
      SELECT jsonb_agg(jsonb_build_array(
        t."id"::text,
        t."site_id"::text,
        t."order_id"::text,
        t."status"::text,
        t."priority"::text,
        t."created_at"::text
      ))
      FROM "orders" t
      WHERE t."order_id" = ? AND t."site_id" = ?
    ), '[]'::jsonb)
  ),
  'rx', json_build_object(
    'cols', '["id","site_id","order_id","drug_name","quantity"]'::jsonb,
    'rows', COALESCE((
      SELECT jsonb_agg(jsonb_build_array(
        t."id"::text, t."site_id"::text, t."order_id"::text, t."drug_name"::text, t."quantity"::text
      ))
      FROM "rx" t
      WHERE t."order_id" = ? AND t."site_id" = ?
    ), '[]'::jsonb)
  )
)::text AS bundle
```

For the IN clause variant (Phase 2/3), the WHERE is: `t."rx_id" IN (?, ?, ?)` with no site_id filter.

**Column names** are loaded from information_schema.columns (ORDER BY ordinal_position) and cached in ConcurrentHashMap per table.

**Helper methods in base class:**
- `getColumns(tableName)` — cached column list from information_schema
- `buildQueryWithSingleKey()` — builds json_build_object with WHERE key = ? AND site_id = ?
- `buildQueryWithInClause()` — builds json_build_object with WHERE key IN (?,?,?)
- `appendTableSubquery()` — shared logic for building each table's subquery (cols jsonb + rows with ::text casting)
- `executeAndParse()` — executes query, parses result JSON into Map<String, String>
- `emptyResultForTables()` — returns {"cols":[...],"rows":[]} for each table when no data
- `clearCache()` — clears column cache

### OrderGroupFetcher, RxGroupFetcher, ContainerGroupFetcher

Each is a @Component that extends TableGroupFetcher. Each is very small:
OrderGroupFetcher:
getGroupName() → "ORDER"
fetch(tables, orderId, siteId) → calls fetchBySingleKey(tables, "order_id", orderId, siteId)
RxGroupFetcher:
getGroupName() → "RX"
fetch(tables, rxIds) → calls fetchByMultipleIds(tables, "rx_id", rxIds)
ContainerGroupFetcher:
getGroupName() → "CONTAINER"
fetch(tables, containerIds) → calls fetchByMultipleIds(tables, "container_id", containerIds)

### BundleBuilder (orchestrator)

Constructor injects: EventTableMapping, OrderGroupFetcher, RxGroupFetcher, ContainerGroupFetcher.

Constants for bridge table names (configurable):
- RX_BRIDGE_TABLE = "rx" (ORDER-group table whose "id" provides rx_ids)
- RX_BRIDGE_ID_COLUMN = "id"
- CONTAINER_BRIDGE_TABLE = "container" (ORDER-group table whose "id" provides container_ids)
- CONTAINER_BRIDGE_ID_COLUMN = "id"

`buildBundle(orderId, siteId, event)` method:

1. Get EventTables from EventTableMapping (has orderTables, rxTables, containerTables lists)
2. Phase 1: orderGroupFetcher.fetch(orderTables, orderId, siteId) → put all into allResults map
3. If event has RX tables: extract rx_ids = orderGroupFetcher.extractIds(allResults.get("rx"), "id")
4. If event has CONTAINER tables: extract containerIds = orderGroupFetcher.extractIds(allResults.get("container"), "id")
5. Phase 2 (if RX tables exist): rxGroupFetcher.fetch(rxTables, rxIds) → put into allResults
6. Phase 3 (if CONTAINER tables exist): containerGroupFetcher.fetch(containerTables, containerIds) → put into allResults
7. Assemble allResults into JSON string: {"table1":{...},"table2":{...},...}
8. Log: orderId, event, table count, phase count, elapsed ms

### EventTableMapping

Loads from sync_event_table_config with table_group column. Returns EventTables object that has:
- getOrderTables() → List<String>
- getRxTables() → List<String>
- getContainerTables() → List<String>
- hasRxTables() → boolean
- hasContainerTables() → boolean

EventTables has a Builder that accepts addTable(tableName, group) and routes to the correct list based on group string ("ORDER", "RX", "CONTAINER").

Cache with AtomicReference, refresh every 5 minutes via @Scheduled, keep old values on DB failure.
