
Create a Spring Boot 3.2 service called "order-sync-consumer" that consumes Kafka messages and dynamically upserts data into a GCP PostgreSQL database.

## Context
- 40 pharmacy edge sites publish order sync messages to a Kafka topic
- Each message contains data for one order across multiple tables
- This consumer runs on GCP and upserts into central Cloud SQL PostgreSQL
- The consumer must handle ANY table dynamically — no table-specific code
- The consumer does NOT know how the publisher works internally

## Kafka Message Structure

Each Kafka message is a JSON string with this structure:
```json
{
  "messageId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "orderId": "ORD-500",
  "siteId": "SITE_001",
  "eventType": "ORDER_PICKED",
  "bundleJson": "<escaped JSON string containing the table data - see below>",
  "createdAt": "2026-02-22T10:30:05Z",
  "outboxId": 42,
  "retryCount": 0
}
```

- messageId: unique UUID per message (for idempotency/logging)
- orderId: the order this data belongs to
- siteId: which pharmacy site sent this
- eventType: business event name (for logging only — consumer doesn't need to act on this)
- bundleJson: a JSON STRING (escaped) containing the actual table data — see format below
- createdAt: when the message was created at the source
- outboxId: internal ID from the source (for logging only)
- retryCount: how many times the source retried (for logging only)

Kafka message key: "{siteId}_{orderId}" (e.g., "SITE_001_ORD-500"). This ensures all events for the same order go to the same partition and are processed in order.

## Bundle Format (inside bundleJson after parsing)

When you parse the bundleJson string, you get a JSON object where each key is a table name and each value is an object with "cols" (column names) and "rows" (data):
```json
{
  "orders": {
    "cols": ["id", "site_id", "order_id", "status", "priority", "created_at", "updated_at"],
    "rows": [
      ["1", "SITE_001", "ORD-500", "PICKED", "1", "2026-02-22 10:00:00+00", "2026-02-22 10:30:00+00"]
    ]
  },
  "rx": {
    "cols": ["id", "site_id", "order_id", "rx_number", "drug_name", "quantity", "status", "created_at"],
    "rows": [
      ["100", "SITE_001", "ORD-500", "RX-001", "Lisinopril", "30", "ACTIVE", "2026-02-22 10:00:00+00"],
      ["101", "SITE_001", "ORD-500", "RX-002", "Metformin", "60", "ACTIVE", "2026-02-22 10:00:00+00"]
    ]
  },
  "container": {
    "cols": ["id", "site_id", "order_id", "container_type", "barcode", "status", "created_at"],
    "rows": [
      ["300", "SITE_001", "ORD-500", "VIAL", "CTN-12345", "PICKED", "2026-02-22 10:30:00+00"]
    ]
  },
  "container_content": {
    "cols": ["id", "site_id", "order_id", "container_id", "rx_id", "quantity", "created_at"],
    "rows": [
      ["400", "SITE_001", "ORD-500", "300", "100", "30", "2026-02-22 10:30:00+00"],
      ["401", "SITE_001", "ORD-500", "300", "101", "60", "2026-02-22 10:30:00+00"]
    ]
  },
  "order_history": {
    "cols": ["id", "site_id", "order_id", "action", "performed_by", "created_at"],
    "rows": [
      ["50", "SITE_001", "ORD-500", "PICKED", "USER_01", "2026-02-22 10:30:00+00"]
    ]
  },
  "rx_staging": {
    "cols": ["id", "site_id", "order_id", "status"],
    "rows": []
  }
}
```

Key points about this format:
- "cols": array of column names. This tells you which column each position in the row maps to. cols[0] = first value in each row, cols[1] = second value, etc.
- "rows": array of arrays. Each inner array is one database row. ALL values are TEXT strings, formatted by the source PostgreSQL database using ::text casting. Null database values appear as JSON null.
- Tables with no data have an empty rows array. Skip these.
- The bundle may contain any number of tables (typically 5-13 depending on the event).

## How to Upsert (critical design — follow exactly)

Since all values arrive as TEXT strings that were formatted by a PostgreSQL database, the consumer should send them back to PostgreSQL as TEXT with ::type casting. PostgreSQL parses its own text format — this is guaranteed to work with zero Java type conversion.

### Step 1: Read column TYPES and PRIMARY KEYS from the consumer's own database (not from the message)

Column types query (cache the results per table in ConcurrentHashMap):
```sql
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = ? AND table_schema = 'public'
```
Returns: {id → bigint, site_id → character varying, status → character varying, created_at → timestamp with time zone, ...}

Primary key query (cache the results per table in ConcurrentHashMap):
```sql
SELECT a.attname
FROM pg_index i
JOIN pg_attribute a ON a.attrelid = i.indrelid AND a.attnum = ANY(i.indkey)
WHERE i.indrelid = ?::regclass AND i.indisprimary
ORDER BY array_position(i.indkey, a.attnum)
```
Returns: [id, site_id]

### Step 2: For each table in the bundle, build an INSERT ... ON CONFLICT with ::type casts

For the "orders" table from the sample above:
- cols from message: [id, site_id, order_id, status, priority, created_at, updated_at]
- types from DB: {id=bigint, site_id=character varying, order_id=character varying, status=character varying, priority=integer, created_at=timestamp with time zone, updated_at=timestamp with time zone}
- pk from DB: [id, site_id]

Generated SQL:
```sql
INSERT INTO orders (id, site_id, order_id, status, priority, created_at, updated_at)
VALUES (?::bigint, ?::varchar, ?::varchar, ?::varchar, ?::integer, ?::timestamptz, ?::timestamptz)
ON CONFLICT (id, site_id) DO UPDATE SET
  order_id = EXCLUDED.order_id,
  status = EXCLUDED.status,
  priority = EXCLUDED.priority,
  created_at = EXCLUDED.created_at,
  updated_at = EXCLUDED.updated_at
```

PK columns (id, site_id) are excluded from the UPDATE SET clause.
If ALL columns are PK columns, use ON CONFLICT ... DO NOTHING.

### Step 3: Send ALL values as plain Java String

For the row ["1", "SITE_001", "ORD-500", "PICKED", "1", "2026-02-22 10:00:00+00", "2026-02-22 10:30:00+00"]:
```java
Object[] params = {"1", "SITE_001", "ORD-500", "PICKED", "1", "2026-02-22 10:00:00+00", "2026-02-22 10:30:00+00"};
// All strings. No Integer, no Timestamp, no BigDecimal.
// The ?::bigint in SQL makes PostgreSQL parse "1" into bigint.
// The ?::timestamptz makes PostgreSQL parse "2026-02-22 10:00:00+00" into timestamp.
```

For null values (JSON null in the row array): pass Java null.

Use jdbcTemplate.batchUpdate(sql, listOfParamArrays) for efficiency.

### Step 4: mapToSqlType() mapping

Map information_schema data_type values to PostgreSQL cast types:
- "integer" → "integer"
- "smallint" → "smallint"  
- "bigint" → "bigint"
- "numeric" → "numeric"
- "decimal" → "numeric"
- "real" → "real"
- "double precision" → "double precision"
- "boolean" → "boolean"
- "character varying" → "varchar"
- "character" → "char"
- "text" → "text"
- "timestamp with time zone" → "timestamptz"
- "timestamp without time zone" → "timestamp"
- "date" → "date"
- "time with time zone" → "timetz"
- "time without time zone" → "time"
- "uuid" → "uuid"
- "jsonb" → "jsonb"
- "json" → "json"
- "bytea" → "bytea"
- "inet" → "inet"
- anything else → "text"

### Validation before upserting each table:
1. Table name must be in ALLOWED_TABLES set (security against SQL injection)
2. tableData must have "cols" array and "rows" array
3. cols must not be empty
4. Every column in cols must exist in the types map from the database (if not, log error: "Column 'X' from message not found in target table — schema mismatch", skip this table)
5. Every PK column must be present in cols (if not, log error, skip table)
6. Every row array must have same length as cols array (if not, log warning, skip that row)

## ALLOWED_TABLES set
orders, patient, rx, order_details, rx_details, order_comment, order_history, order_hold_exclusion, tote, tote_history, rx_staging, rx_loc, rx_history, container, container_content, container_history, container_content_history

## Component Details

### KafkaConsumerConfig
- MANUAL_IMMEDIATE ack mode: offset committed only after successful database upsert
- ErrorHandlingDeserializer wrapping StringDeserializer: if a message can't be deserialized, log and skip it (acknowledge), don't retry. This prevents poison pill messages from blocking the consumer.
- DefaultErrorHandler with FixedBackOff(5000L, 3L): retry processing errors 3 times with 5-second delay. After 3 failures, skip the message.
- Add JsonProcessingException as not-retryable exception
- max.poll.records=10, enable.auto.commit=false, session.timeout.ms=30000
- Concurrency configurable: ${order-sync.consumer.concurrency:3}

### OrderSyncListener (@KafkaListener)
- Topic: ${order-sync.consumer.topic:order-sync-events}
- Group: ${spring.kafka.consumer.group-id:order-sync-consumer-group}
- Method receives: ConsumerRecord<String, String> record, Acknowledgment acknowledgment
- Step 1: If record.value() is null or blank → log warning with partition/offset, acknowledge, return
- Step 2: Deserialize value to SyncMessage using ObjectMapper (with JavaTimeModule registered). If JsonProcessingException → log error with partition/offset, acknowledge (skip poison pill), return
- Step 3: Validate message.orderId not null/blank, message.siteId not null/blank, message.bundleJson not null/blank. If any invalid → log error, acknowledge, return
- Step 4: Call syncProcessingService.processMessage(message)
- Step 5: On success → acknowledgment.acknowledge(), log info with orderId, siteId, eventType, partition, offset, result
- Step 6: On exception → do NOT acknowledge. Log error. Throw RuntimeException so the error handler retries.

### SyncProcessingService
- processMessage(SyncMessage message) annotated with @Transactional
- Validate message fields
- Parse bundleJson string into JsonNode using ObjectMapper
- Record start time for elapsed calculation
- Iterate through table names: bundle.fieldNames()
- For each table:
  - Skip if not in ALLOWED_TABLES
  - Get tableData = bundle.get(tableName)
  - Skip if tableData is null or doesn't have "cols" or "rows"
  - Skip if rows array is empty
  - Call dynamicUpsertService.upsertTable(tableName, tableData)
  - Collect result in Map<String, Integer>
- Log: orderId, siteId, eventType, number of tables, total rows, elapsed ms
- Return the result map
- If any table upsert throws an exception, the @Transactional rolls back ALL tables for this message (all-or-nothing per order event)
